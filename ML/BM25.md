# BM25 — Okapi Best Match 25

> [!summary]
> BM25 (Best Match 25) is a **probabilistic ranking function** used in information retrieval to score the relevance of documents to a given search query. It is the backbone of most classical keyword-based search engines and remains a strong baseline even in the era of neural IR.

---

## 🧠 Intuition

BM25 answers the question:

> _"Given a query with terms $q_1, q_2, \ldots, q_n$, how relevant is document $D$?"_

It improves on the simpler TF-IDF by addressing two key saturation problems:

1. **Term frequency saturation** — seeing a word 100× shouldn't be 100× better than seeing it 10×.
2. **Document length normalization** — a long document matches more terms just because it's long.

---

## 📐 Formula

$$ \text{BM25}(D, Q) = \sum_{i=1}^{n} \text{IDF}(q_i) \cdot \frac{f(q_i, D) \cdot (k_1 + 1)}{f(q_i, D) + k_1 \cdot \left(1 - b + b \cdot \frac{|D|}{\text{avgdl}}\right)} $$

Where:

| Symbol            | Meaning                                                   |
| ----------------- | --------------------------------------------------------- |
| $f(q_i, D)$       | Term frequency of $q_i$ in document $D$                   |
| $\|D\|$           | Length of document $D$ (in words)                         |
| $\text{avgdl}$    | Average document length in the corpus                     |
| $k_1$             | Term frequency saturation parameter (typically `1.2–2.0`) |
| $b$               | Length normalization parameter (typically `0.75`)         |
| $\text{IDF}(q_i)$ | Inverse Document Frequency of $q_i$                       |

### IDF Term

$$ \text{IDF}(q_i) = \ln\left(\frac{N - n(q_i) + 0.5}{n(q_i) + 0.5} + 1\right) $$

| Symbol   | Meaning                              |
| -------- | ------------------------------------ |
| $N$      | Total number of documents            |
| $n(q_i)$ | Number of documents containing $q_i$ |

---

## ⚙️ Key Parameters

### `k1` — TF Saturation

- Controls how quickly term frequency saturates.
- **Low $k_1$** (→ 0): Binary presence/absence, TF ignored entirely.
- **High $k_1$** (→ ∞): Raw TF, no saturation.
- Typical value: **1.2 to 2.0**

### `b` — Length Normalization

- Controls how much document length affects scoring.
- **$b = 0$**: No length normalization.
- **$b = 1$**: Full normalization (score proportional to density).
- Typical value: **0.75**

> [!tip] Tuning Tip Start with `k1=1.5, b=0.75`. For short-form corpora (tweets, titles), lower `b` toward 0.3.

---

## 🔄 BM25 vs TF-IDF

| Property             | TF-IDF        | BM25                  |
| -------------------- | ------------- | --------------------- |
| TF saturation        | None (linear) | Logarithmic plateau   |
| Length normalization | Basic         | Tunable via `b`       |
| IDF formula          | $\log(N/n)$   | Smoothed, more stable |
| Performance          | Good baseline | Consistently better   |

---

## 🔬 Variants

### BM25+

Addresses a flaw where documents with zero TF for a term get no contribution. Adds a lower-bound delta ($\delta$) to the TF component.

$$ \frac{f(q_i, D) \cdot (k_1 + 1)}{f(q_i, D) + k_1(\ldots)} + \delta $$

### BM25F (Field-based)

Extends BM25 to structured documents with multiple fields (title, body, URL). Each field gets its own weight and length normalization.

$$ \tilde{f}(q_i, D) = \sum_{\text{field}} w_f \cdot \frac{f_f(q_i, D)}{1 - b_f + b_f \cdot \frac{|D_f|}{\text{avgdl}_f}} $$

### BM25L

Another variant that further corrects TF normalization for long documents.

## 🛠️ Implementation

### Python (rank_bm25 library)

```python
from rank_bm25 import BM25Okapi

corpus = [
    "the dog sat on the mat",
    "the cat sat on the mat",
    "the quick brown fox",
]

tokenized = [doc.split() for doc in corpus]
bm25 = BM25Okapi(tokenized, k1=1.5, b=0.75)

query = "cat mat".split()
scores = bm25.get_scores(query)
top_docs = bm25.get_top_n(query, corpus, n=2)
```

### Elasticsearch

BM25 is the **default similarity** in Elasticsearch since version 5.0.

```json
PUT /my_index
{
  "settings": {
    "similarity": {
      "custom_bm25": {
        "type": "BM25",
        "k1": 1.2,
        "b": 0.75
      }
    }
  }
}
```

## 🔗 BM25 in Modern IR Pipelines

BM25 is widely used as the **first-stage retriever** in hybrid and neural search pipelines:

```
Query
  │
  ▼
BM25 (fast, lexical)  ──────────────────────────────────┐
  │                                                       │
  │  top-k candidates                                     │
  ▼                                                       │
Dense Retriever (ANN / FAISS)                            │
  │                                                       │
  │  top-k candidates                                     ▼
  └──────────────────────► Reciprocal Rank Fusion (RRF)
                                    │
                                    ▼
                              Re-ranker (cross-encoder)
                                    │
                                    ▼
                              Final Ranked Results
```

### Why keep BM25 in a neural pipeline?

- **Lexical precision**: Exact keyword matches (product codes, names, acronyms) that dense models miss.
- **Speed**: Inverted index lookup is $O(\text{query terms})$, much faster than ANN search.
- **No training required**: Works zero-shot on any domain.
- **Complementary recall**: Dense and sparse retrievers catch different relevant docs.

---

## ✅ Strengths

- Simple, interpretable, fast
- Strong baseline — hard to beat on keyword-heavy queries
- Domain-agnostic (no training needed)
- Works well on exact-match needs (IDs, codes, rare terms)

## ❌ Weaknesses

- **Vocabulary mismatch**: "car" ≠ "automobile" (no semantic understanding)
- **No context**: Word order and meaning ignored
- **Rare term sensitivity**: IDF heavily penalizes terms in many docs
- **No learning**: Cannot adapt to user behavior or domain signals
